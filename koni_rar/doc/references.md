# koni_rar — reference materials (per §13.7 and doc/rar-provenance.md)

Every reference consulted for the RAR implementation, recorded at first
use. **No unrar source and no GPL/LGPL source was consulted.**

| Reference | License / status | Used for |
| --- | --- | --- |
| libarchive `archive_read_support_format_rar5.c` (Grzegorz Antoniak, 2018) | BSD-2-Clause (notice retained in `NOTICE`) | RAR5 block/header layout, compressed-block bitstream structure, Huffman table encoding, filter semantics — consulted and adapted |
| libarchive `archive_read_support_format_rar.c` (Tim Kientzle 2003–2007, Andres Mejia 2011) | BSD-2-Clause (notice retained in `NOTICE`) | RAR4 (v1.5) block/header layout and the method-29 (v29) LZSS+Huffman unpack (block tables, length/distance decoding, short/long LZ) — consulted and adapted. **RarVM standard filters** (`rar4_filters.dart`): the filter-record reader (`read_filter`/`parse_filter`/`compile_program`), the memory bit-reader + vint (`membr_*`), the fingerprint dispatch, and the delta/E8/E8E9/RGB/audio `execute_filter_*` algorithms — consulted and adapted. **PPMd variant H glue** (`rar4_ppmd.dart` / `rar4_decoder.dart`): the RAR-specific range decoder (`PpmdRAR_RangeDec_Init`/`Normalize`/`GetThreshold`/`Decode`/`DecodeBit`), the PPM block-header parse (max-order / sub-alloc size / reset flags), and the escape-char symbol dispatch in `read_next_symbol`/`expand` (new-table, EOF, VM-filter, LZ distance/length, reused distance) — consulted and adapted. A *generic* RarVM bytecode interpreter (for non-standard programs) is still **not** implemented — the only interpreter reference is GPL unrar, so custom filters (including those reached via a PPMd escape) stay a typed error. |
| libarchive `archive_ppmd7.c` / `archive_ppmd7_private.h` / `archive_ppmd_private.h` (Igor Pavlov, 2010-03-12; based on PPMd var.H 2001 by Dmitry Shkarin) | **Public domain** (stated in the file headers; vendored by libarchive from the LZMA SDK) | The PPMd variant H model itself (`rar4_ppmd.dart`): the unit sub-allocator (`Ppmd7_Construct`/`Alloc`/`InsertNode`/`RemoveNode`/`SplitBlock`/`GlueFreeBlocks`/`AllocUnits`/`ShrinkUnits`/`ExpandUnits`), the suffix-linked context tree, `CreateSuccessors`/`UpdateModel`/`Rescale`, SEE (`Ppmd_See_Update`), the `BinSumm`/`NS2Indx`/`NS2BSIndx`/`HB2Flag` tables, and `Ppmd7_DecodeSymbol`/`Ppmd7_MakeEscFreq` — consulted and adapted. Public-domain source, so no copyright notice is required, but attribution is recorded here and in `NOTICE`. |
| rarfile project documentation (`rarfile.readthedocs.io`) — format notes | MIT-licensed project docs | RAR5 container field layout cross-check; RAR5 encryption record + KDF description (P3-4) |
| Go `rardecode` (`github.com/nwaples/rardecode`, `archive50.go`, `decode29.go`, `decode29_ppm.go`, `ppm_model.go`, `decode20.go`, `decode20_lz.go`, `huffman.go`) | BSD-2-Clause | (a) RAR5 encrypted-header (`-hp`) block framing cross-check: per-block clear IV + AES-256-CBC header padded to 16, block key covers headers only (data uses the per-file record), and the file-record flag split (bit 0 password-check vs bit 1 "use MAC"/CRC-tweak). libarchive's `rar5.c` has **no** crypto, so it could not serve here. (b) **Solid RAR4 PPMd** control flow, which libarchive cannot supply (it supports no solid RAR at all): each solid file is its own PPMd block ending with an escape-code-2 marker (`endOfBlockAndFile`), the shared model and the RAR-block escape symbol carry across files (the escape is reset only by flag 0x40, not per block — libarchive's per-block `escape = 2` is wrong for the solid carry-over it never exercises), and the range decoder re-initialises per block. The Ppmd7 model + range-decoder arithmetic in `ppm_model.go` cross-checked against the same public-domain codec; consulted and adapted for the solid framing only. (c) **RAR 2.0/2.6 (unpack v20/v26) LZ** decoder (`rar20_decoder.dart`): the block header (audio bit + keep-table bit), the 19-symbol pre-code / code-length table reader, the main/offset/length Huffman sizes (298/48/28) and symbol dispatch (literal, reused offset, length-from-history, short offset, end-of-block, offset) — consulted and adapted. `decode20`'s multimedia/audio predictor was **not** adopted: it mis-decodes (rardecode itself fails the audio fixture's CRC vs unrar), so the audio block stays a typed error. The canonical Huffman assignment matches `huffman.go` (verified byte-exact). |
| RARLAB RAR5 archive format technote (`www.rarlab.com/technote.htm`) | public format documentation | RAR5 encryption record fields, AES-256-CBC, PBKDF2-HMAC-SHA256 key/IV/check derivation (P3-4) |
| RAR3/RAR4 encryption format notes (rarfile docs + published clean-room analyses) | MIT docs / public analyses | RAR4 encryption: SHA-1 KDF (0x40000 rounds), header salt, AES-128-CBC key/IV derivation (P3-5) — verified empirically against rar 6.24 output |
| `rar`/`unrar` binaries (RARLAB) | proprietary tools, used as black boxes | fixture generation and behavioral ground truth (§13.3), incl. empirical verification of the P3-4 KDF/tweaked-CRC against authored fixtures — never their source |
