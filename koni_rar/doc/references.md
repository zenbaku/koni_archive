| libarchive `archive_read_support_format_rar.c` (Tim Kientzle 2003–2007, Andres Mejia 2011) | BSD-2-Clause (notice retained in `NOTICE`) | RAR4 (v1.5) block/header layout and the method-29 (v29) LZSS+Huffman unpack (block tables, length/distance decoding, short/long LZ) — consulted and adapted. **RarVM standard filters** (`rar4_filters.dart`): the filter-record reader (`read_filter`/`parse_filter`/`compile_program`), the memory bit-reader + vint (`membr_*`), the fingerprint dispatch, and the delta/E8/E8E9/RGB/audio `execute_filter_*` algorithms — consulted and adapted. **PPMd variant H glue** (`rar4_ppmd.dart` / `rar4_decoder.dart`): the RAR-specific range decoder (`PpmdRAR_RangeDec_Init`/`Normalize`/`GetThreshold`/`Decode`/`DecodeBit`), the PPM block-header parse (max-order / sub-alloc size / reset flags), and the escape-char symbol dispatch in `read_next_symbol`/`expand` (new-table, EOF, VM-filter, LZ distance/length, reused distance) — consulted and adapted. | rarfile project documentation (`rarfile.readthedocs.io`) — format notes | MIT-licensed project docs | RAR5 container field layout cross-check; RAR5 encryption record + KDF description (P3-4) |
| Go `rardecode` (`github.com/nwaples/rardecode`, `archive15.go`, `archive50.go`, `decrypt_reader.go`, `vm.go`, `filters.go`, `bit_reader.go`, `decode29.go`, `decode29_ppm.go`, `ppm_model.go`, `decode20.go`, `decode20_lz.go`, `huffman.go`) | BSD-2-Clause | (a) RAR5 encrypted-header (`-hp`) block framing cross-check: per-block clear IV + AES-256-CBC header padded to 16, block key covers headers only (data uses the per-file record), and the file-record flag split (bit 0 password-check vs bit 1 "use MAC"/CRC-tweak). libarchive's `rar5.c` has **no** crypto, so it could not serve here. (a2) **RAR3/4 encrypted-header (`-hp`) framing** (`archive15.go` `readBlockHeader`/`parseArcBlock` + `decrypt_reader.go` `newAesDecryptReader`): the marker + main header stay plaintext (the main header carries the `MHD_PASSWORD` 0x0080 flag); every following block is `salt[8] · AES-128-CBC(header padded to 16)` with the cipher re-initialised per block from the salt-derived IV (CBC chains only within a block); the salt is repeated verbatim before each block; file *data* between headers uses each file's own SALT-flag key (the existing `-p` path); and the 16-bit header CRC (`crc32(header[2:size]) & 0xFFFF`) is the wrong-password signal (RAR3/4 has no password-check value). Adapted for `rar4_container.dart`'s `_parseRar4EncryptedHeaders`; libarchive's RAR4 reader has no crypto. Empirically pinned byte-exact against rar 6.24 `-ma4 -hp` fixtures (header CRCs and decrypted content reproduce; `unrar` is the oracle). (b) **Solid RAR4 PPMd** control flow, which libarchive cannot supply (it supports no solid RAR at all): each solid file is its own PPMd block ending with an escape-code-2 marker (`endOfBlockAndFile`), the shared model and the RAR-block escape symbol carry across files (the escape is reset only by flag 0x40, not per block — libarchive's per-block `escape = 2` is wrong for the solid carry-over it never exercises), and the range decoder re-initialises per block. The Ppmd7 model + range-decoder arithmetic in `ppm_model.go` cross-checked against the same public-domain codec; consulted and adapted for the solid framing only. (c) **RAR 2.0/2.6 (unpack v20/v26) LZ** decoder (`rar20_decoder.dart`): the block header (audio bit + keep-table bit), the 19-symbol pre-code / code-length table reader, the main/offset/length Huffman sizes (298/48/28) and symbol dispatch (literal, reused offset, length-from-history, short offset, end-of-block, offset) — consulted and adapted. `decode20`'s multimedia/audio predictor was **not** adopted: it mis-decodes (rardecode itself fails the audio fixture's CRC vs unrar), so the audio block stays a typed error. The canonical Huffman assignment matches `huffman.go` (verified byte-exact). (d) **Generic RarVM interpreter** (`rar4_vm.dart`, R6): the machine structure and opcode/flag semantics (`vm.go` — registers, memory, C/Z/S flags, the ~40 `ops` table, `readCommands`/`decodeArg`/`fixJumpOp` bit decoding), the operand addressing modes, and the filter global-block wiring (`filters.go` `vmFilter.execute` — fixed globals at `0x3C000`, user global + static data, register setup r3..r7, and reading the output region from `vg[0x1c]`/`vg[0x20]` with the `vg[0x30]` persistent-global carry) — consulted and adapted; the MSB-first bit order + `readUint32` variable number match `bit_reader.go`. libarchive has no VM interpreter, so it could not serve. Verified byte-exact by running the four *standard* programs (real RarVM bytecode) through the interpreter to the same fixture CRCs (`unrar`-checked). |
